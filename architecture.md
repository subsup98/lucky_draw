# 시스템 구조 초안 (architecture.md)

## 1. 목표
이 문서는 이치방쿠지 서비스의 기본 시스템 구조를 설명한다.

## 2. 구성 요소
- 사용자 웹 프론트엔드
- 관리자 웹 프론트엔드
- API 서버
- 인증/권한 처리 모듈
- 결제 연동 모듈
- 추첨 엔진
- 재고 관리 모듈
- 주문/배송 모듈
- 알림 모듈
- 데이터베이스
- 로그/모니터링 시스템
- 이미지 저장소

---

## 3. 기본 흐름

### 3.1 사용자 구매 흐름
1. 사용자가 쿠지 상세 페이지에 진입한다.
2. 구매 횟수를 선택한다.
3. 주문을 생성한다.
4. 결제를 진행한다.
5. 서버가 결제 성공을 검증한다.
6. 서버가 추첨을 실행한다.
7. 재고를 차감하고 결과를 저장한다.
8. 사용자에게 결과를 반환한다.
9. 마이페이지 및 주문 내역에 결과가 반영된다.

### 3.2 관리자 운영 흐름
1. 운영자가 관리자 페이지에 로그인한다.
2. 쿠지 이벤트를 생성한다.
3. 경품 등급 및 수량을 등록한다.
4. 판매 기간과 제한 수량을 설정한다.
5. 주문 및 배송 상태를 관리한다.
6. 이슈 발생 시 로그를 확인하고 처리한다.

## 6. 핵심 설계 포인트

### 6.1 추첨 정합성
- 결제 성공 이후에만 추첨
- 재고 차감과 결과 저장을 하나의 흐름으로 처리
- 동시성 문제 방지

### 6.2 운영 권한 분리
- 일반 운영자와 슈퍼 관리자 분리
- 민감 작업 로그 기록

### 6.3 확장성
- 향후 앱, 다국어, 쿠폰, 포인트 기능 추가 가능 구조
- 외부 결제사/택배사 교체 가능 구조

## 7. 배포 관점 초안
- 프론트엔드: CDN 또는 정적 호스팅
- 백엔드: API 서버 + 배치/워커 가능
- DB: 트랜잭션 지원 RDB 권장
- 스토리지: 상품 이미지 및 첨부 이미지 저장
- 모니터링: 에러 로그, 성능 지표, 결제 실패율 확인
