# 보안 문서 (security.md)

## 1. 목적
이 문서는 이치방쿠지 서비스의 계정, 결제, 추첨, 개인정보, 운영자 권한과 관련된 보안 원칙을 정의한다.

## 2. 보안 목표
- 추첨 결과 조작 방지
- 결제 위변조 방지
- 개인정보 유출 방지
- 관리자 권한 오남용 방지
- 봇/매크로/부정 이용 방지
- 장애 및 사고 발생 시 추적 가능성 확보

---

## 3. 핵심 보안 원칙

### 3.1 서버 측 추첨
- 추첨 결과는 서버에서만 생성한다.
- 클라이언트는 결과 연출만 담당하고 결과를 결정하지 않는다.
- 추첨 실행과 재고 차감은 하나의 일관된 처리 흐름으로 묶어야 한다.

### 3.2 결제 후 확정
- 결제 성공 검증 전에는 추첨 결과를 확정하지 않는다.
- 결제 콜백과 내부 주문 상태를 일치시켜야 한다.
- 중복 요청이 와도 동일 주문에 대해 추첨은 1회만 확정되어야 한다.

### 3.3 최소 권한 원칙
- 일반 사용자, 운영자, 슈퍼 관리자의 권한을 분리한다.
- 운영자는 필요한 메뉴만 접근 가능해야 한다.
- 민감 작업은 슈퍼 관리자 또는 별도 승인 절차를 요구할 수 있다.

### 3.4 감사 가능성
- 추첨 실행
- 재고 수정
- 환불 처리
- 권한 변경
- 공지 강제 노출
- 회원 제재
위 항목은 반드시 로그를 남겨야 한다.

---

## 4. 인증 및 계정 보안

### 4.1 로그인
- 비밀번호는 안전한 해시 알고리즘으로 저장한다.
- 로그인 실패 횟수 제한 또는 지연 처리가 필요하다.
- 장시간 미사용 세션은 만료되어야 한다.

### 4.2 계정 보호
- 이메일 인증 또는 소셜 계정 검증을 지원한다.
- 의심스러운 로그인 시 추가 확인 절차를 둘 수 있다.
- 관리자 계정은 2단계 인증 적용을 권장한다.

### 4.3 비밀번호 정책
- 최소 길이 요구
- 너무 단순한 문자열 제한
- 비밀번호 재설정 토큰 만료시간 적용

---

## 5. 결제 보안

### 5.1 결제 요청
- 결제 금액은 서버 기준으로 계산한다.
- 클라이언트가 전달한 금액을 그대로 신뢰하지 않는다.

### 5.2 결제 검증
- 결제 완료 여부는 결제사 응답과 서버 검증으로 확인한다.
- 취소, 실패, 타임아웃 상태를 구분해 처리한다.

### 5.3 중복 결제 방지
- 주문 단위 고유 키를 사용한다.
- 동일 요청 재시도 시 멱등성 처리가 필요하다.

---

## 6. 추첨 및 재고 보안

### 6.1 재고 정합성
- 이미 소진된 경품이 다시 당첨되면 안 된다.
- 재고 차감과 결과 생성 사이에 불일치가 없어야 한다.

### 6.2 동시성 제어
- 동시에 여러 명이 같은 쿠지를 뽑아도 중복 당첨이 발생하지 않아야 한다.
- 락, 트랜잭션, 큐 등 적절한 방식으로 정합성을 보장한다.

### 6.3 결과 변경 방지
- 생성된 추첨 결과는 운영자가 임의 수정할 수 없어야 한다.
- 예외 처리 시에도 원본 이력은 보존해야 한다.

---

## 7. 개인정보 보호

### 7.1 수집 최소화
- 구매와 배송에 필요한 정보만 수집한다.
- 불필요한 민감정보는 수집하지 않는다.

### 7.2 저장 및 노출
- 주소, 연락처, 이름 등은 접근 권한을 제한한다.
- 관리자 화면에서는 필요한 범위 내에서 마스킹을 적용할 수 있다.

### 7.3 보관 및 파기
- 법적 보관 의무가 지난 개인정보는 파기 또는 비식별 처리한다.
- 탈퇴 사용자의 정보는 정책과 법령에 따라 별도 보관 후 삭제한다.

---

## 8. 운영자 보안

### 8.1 관리자 권한 분리
- 콘텐츠 운영
- 주문/배송 운영
- CS 운영
- 권한/정책 운영
역할을 분리할 수 있어야 한다.

### 8.2 민감 작업 보호
- 재고 대량 수정
- 권한 변경
- 환불 승인
- 정책 변경
위 작업은 별도 로그와 추가 확인을 남겨야 한다.

### 8.3 관리자 접속 기록
- 관리자 로그인 시간
- 접속 IP 또는 디바이스 정보
- 주요 액션 이력
을 기록할 수 있어야 한다.

---

## 9. 악용 방지

### 9.1 봇/매크로 대응
- 비정상 반복 요청을 제한한다.
- 특정 구간에 CAPTCHA 또는 추가 검증을 둘 수 있다.

### 9.2 구매 남용 대응
- 1인당 구매 제한
- 다계정 의심 패턴 감지
- 결제 실패 반복 탐지
- 이상 트래픽 알림

### 9.3 API 보호
- 인증 없는 민감 API 접근 차단
- 속도 제한
- 입력값 검증
- 관리자 API 별도 보호

---

## 10. 비밀정보 관리
- API 키, 결제 키, DB 접속정보는 코드에 하드코딩하지 않는다.
- 환경변수 또는 비밀정보 관리 시스템을 사용한다.
- 운영/개발 환경 비밀값을 분리한다.

---

## 11. 로그 및 모니터링
- 에러 로그, 보안 이벤트 로그, 결제 실패 로그, 추첨 실패 로그를 수집한다.
- 이상 패턴 발생 시 운영자에게 알림을 보낼 수 있어야 한다.
- 로그에는 민감정보 전체를 그대로 남기지 않도록 주의한다.

---

## 12. 사고 대응
- 결제 장애
- 추첨 오류
- 개인정보 노출 의심
- 관리자 계정 탈취 의심
상황별 대응 절차를 별도 운영 문서로 둘 수 있다.
